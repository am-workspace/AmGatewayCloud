# 集成插件网关 vs 分布式微服务网关

## 集成插件网关

**代表产品**：Kepware (PTC)、Matrikon (Honeywell)、Neuron (EMQ)、Ignition (Inductive Automation)、AmGateway

**本质**：一个进程，插件加载驱动，`Channel<T>` 内存管道转发。

### 解决什么问题

- **边缘资源受限** — 工控机/网关盒子通常只有 1-4G 内存、ARM CPU，跑不了微服务全家桶
- **低延迟确定性** — 采集→转发在同一进程内，微秒级，不过网络
- **协议碎片化** — 工业现场几十种协议，用插件扩展是最经济的做法
- **离线自治** — 断网了网关自己照样采集存储，不依赖云端

### 优势

| 优势 | 说明 |
|------|------|
| 部署极简 | 一个 exe 拷过去就跑 |
| 资源占用低 | 百兆内存级 |
| 延迟极低 | 进程内 Channel，不经过网络 |
| 运维简单 | 挂了重启一个进程就行 |
| 离线可用 | 断网不影响采集 |

### 劣势

| 劣势 | 说明 |
|------|------|
| 单点故障 | 进程挂了 = 全部采集中断 |
| 无法水平扩展 | 一个网关的采集能力有上限 |
| 状态不共享 | 两个网关实例看不到彼此的配置和数据 |
| 业务耦合 | 采集和业务逻辑混在一起，越来越重 |

### 典型场景

工厂车间、远程站点、油气井场 — **"边缘"场景**

---

## 分布式/微服务网关

**代表产品**：ThingsBoard、IoTDB + 微服务套件、EMQX + 规则引擎、Azure IoT Hub、AWS IoT Core

**本质**：多个独立服务，通过消息队列/事件总线协作，云/数据中心部署。

### 解决什么问题

- **海量设备接入** — 万级/十万级设备，单机扛不住
- **多业务消费** — 同一份采集数据，报警、历史、分析、工单各自消费
- **高可用** — 一个服务挂了，其他不受影响
- **多租户** — SaaS 化，不同客户隔离

### 优势

| 优势 | 说明 |
|------|------|
| 水平扩展 | 加机器就行，无上限 |
| 故障隔离 | 报警服务挂了，采集不受影响 |
| 独立演进 | 各服务独立部署，互不影响 |
| 多消费方 | 一份数据，N 个服务各取所需 |
| 多租户天然 | 服务层隔离，数据库隔离 |

### 劣势

| 劣势 | 说明 |
|------|------|
| 部署复杂 | 十几个服务 + 中间件 |
| 资源开销大 | 光基础设施就几个 G |
| 延迟增加 | 每经过一个消息队列/网络就多几 ms |
| 运维门槛高 | 需要容器化、监控、链路追踪 |
| 强依赖网络 | 断网了整个系统瘫痪 |

### 典型场景

工业 SaaS 平台、集团级 IoT 平台、设备运维服务商 — **"云端"场景

---

## 核心区别

```
集成插件网关                          分布式微服务网关
解决的是"采集"问题                    解决的是"聚合+业务"问题

  1 台设备 ──► 1 个插件              N 台网关 ──► RabbitMQ ──► 报警服务
  1 台设备 ──► 1 个插件              N 台网关 ──► RabbitMQ ──► 历史服务
       │                                        ──► 分析服务
       ▼                                        ──► 工单服务
   Channel<T>                                  
       │                              关键：数据是一份，消费方各取所需
       ▼                              关键：每个服务可以独立扩缩
   MQTT/InfluxDB
```

- **集成网关做的是"1 对 1"**：一个设备的数据，一条管道，输出到一个地方
- **微服务网关做的是"1 对 N"**：海量设备的数据，一份消息，N 个业务方各取所需

## 工业现实：两者是搭档，不是对手

```
工厂A ──┐                    ┌──► 报警服务
工厂B ──┤  集成网关（边缘）   │
工厂C ──┼──────────────►  云端微服务平台 ──► 历史服务
工厂D ──┤   MQTT/AMQP      │
工厂E ──┘                    └──► 工单服务
       ↑                        ↑
   边缘：采集+缓存+离线       云端：聚合+业务+多租户
   低资源、高可靠             高扩展、高可用
```

Kepware 被卖给 PTC（ThingWorx 云平台）、Neuron 被 EMQ 收购（配合 EMQX 云）—— **边缘+云端是绑定关系**。

## 技术选型对照

| 关注点 | 集成插件网关 | 分布式微服务网关 |
|--------|-------------|----------------|
| 数据管道 | `Channel<T>` 进程内 | RabbitMQ/Kafka 跨进程 |
| 配置存储 | SQLite 本地 | PostgreSQL + Redis 集中 |
| 扩展方式 | 加载插件 DLL | 部署新服务实例 |
| 故障恢复 | 看门狗重启进程 | K8s 自动重启/扩缩 |
| 消费模型 | 1 生产者 → 1 消费者 | 1 生产者 → N 消费者 |
| 多租户 | 不支持 | 服务层 + 数据库隔离 |
| 延迟 | 微秒级 | 毫秒~秒级 |
| 资源 | 百 MB 内存 | 数 GB 内存 |

---

## 我们的项目定位

- **AmGateway** = 边缘层（已做好）— 集成插件网关
- **AmGatewayCloud** = 云端层（正在做）— 分布式微服务平台

两者通过 MQTT/RabbitMQ 解耦，组合为完整的工业物联网解决方案。
